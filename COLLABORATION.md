# 設計師 × AI 協作紀實

> 基於 TrustGuard side project 開發過程
> 展示一位 UI/UX 設計師如何與 Claude Code 協作完成完整前端專案
> 2026 年 3–4 月

---

## 一、協作者背景

| | 人類（我） | AI（Claude Code） |
|---|---|---|
| 角色 | UI/UX 設計師 | 程式實作助手 |
| 強項 | 視覺、互動、文案、使用者邏輯 | 寫 code、除錯、建 Design System、技術判斷 |
| 弱項 | JS 程度有限 | 無法看到實際畫面、不知道我的主觀偏好 |
| 工作環境 | macOS + Claude Code CLI + 瀏覽器預覽 | Terminal + 整個檔案系統的讀寫權限 |

**我沒有把 AI 當工具，而是當成一位「會寫 code 的設計助理」**。分工的前提是我主導設計方向，AI 負責把想法變成跑得起來的 code。

---

## 二、分工模型

```
┌─────────────────────────┐      ┌─────────────────────────┐
│         我              │      │       Claude           │
│  （設計師 / 決策者）     │      │   （實作 / 執行者）     │
├─────────────────────────┤      ├─────────────────────────┤
│ 定義「要做什麼」         │ ───▶ │ 決定「怎麼做」          │
│ 判斷視覺對不對            │      │ 寫程式碼                 │
│ 決定方向                 │      │ 建議技術選型              │
│ 最後 say yes/no         │ ◀─── │ 提出可選方案              │
└─────────────────────────┘      └─────────────────────────┘
```

**重點**：我不會寫規格書（spec），只給「目標」。AI 會提議具體方案，我看到之後再說哪裡要改。

## 三、一天的典型工作流程

```
09:00  我：「今天想做反思頁，要有 trade-offs、學到什麼、下次會改什麼」
        ↓
09:01  AI：提案 4 個區塊 + 4 個子元件，開始寫檔案
        ↓
09:10  我：瀏覽器打開看效果 → 發現 nav bar 沒有 Reflection
        ↓
09:11  我：「nav bar 沒顯示出來 reflection，再檢查一下有啥問題」
        ↓
09:13  AI：找到 NAV 陣列漏掉的地方，補上 + 加手機 shortLabel
        ↓
09:15  我：好 → git commit + push
```

**觀察**：
- 我不讀 AI 寫的 code 細節，只看「跑起來對不對」
- 發現問題時描述「現象」不描述「原因」
- AI 的工作是「把現象翻譯回 code 層級的問題」

## 四、我的溝通模式

### 4.1 訊息偏短、目標導向

| 訊息類型 | 例子 |
|---|---|
| 給任務 | 「做一個反思頁」 |
| 指正 | 「nav bar 沒顯示出來 reflection」 |
| 同意 | 「好」 |
| 比較 | 「這個適合部署到 Zeabur 還是 Cloudflare？」 |
| 委託決定 | 「你幫我選一個最適合的」 |
| 情報輸入 | 「但是 Vercel 最近好像被駭客入侵」 |

**不常出現的訊息類型**：
- 規格書
- 詳細技術要求
- 「請用 React Context 來管理...」這種指定實作方式的訊息

### 4.2 我信任 AI 的判斷，但保留否決權

例子 1 — 部署平台：
> 我：「你幫我選一個最適合的而且要免費，然後部署上去」
> AI：推薦 Vercel（技術上最合適）
> 我：「但是 Vercel 最近好像被駭客入侵」
> AI：立刻改提案 Cloudflare Pages
> 我：「好」

**這個模式的重點**：
- 我放權讓 AI 做技術決定 → 加快速度
- 但當我有 AI 不知道的資訊（安全新聞），我會否決
- AI 不堅持自己的提案，會重新考量

### 4.3 視覺優先的回饋

我幾乎不會在 code 層級回饋。我的回饋都是：
- 「這邊太擠」
- 「手機版看不到」
- 「字太小」
- 「顏色不對」

AI 的工作是把這些視覺語言翻譯成 Tailwind class、CSS 變數、甚至重構元件。

## 五、AI 讓我驚艷的幾個時刻

### 5.1 主動提起被我忽略的議題

我只說「做反思頁」，AI 主動問：「要不要放一個連回首頁的 CTA？」
→ 這是我原本沒想到的，加上去之後整頁完整度提升。

### 5.2 技術選型會把我的限制列出來

我說「我要免費部署」，AI 不只推薦一個選項，而是：
1. 比較 Vercel / Cloudflare / Zeabur 的優缺點
2. 說明為什麼推薦某一個
3. 列出需要注意的事（例如 Vercel 冷啟動、Cloudflare 靜態限制）

→ 讓我即使信任它的判斷，也能理解理由。

### 5.3 除錯比我還仔細

OG image 沒辦法正常顯示時，AI 主動：
1. 用 `file` 指令確認 PNG 真的產生了
2. 檢查 build log 裡的錯誤
3. 發現是 `Content-Type` 沒設對（檔名沒副檔名）
4. 寫 `public/_headers` 修正

這些我完全做不到，但 AI 做了之後會用「我聽得懂的話」解釋為什麼。

## 六、AI 會犯的錯 / 需要人類把關的地方

### 6.1 看不到畫面

AI 寫完 code 之後，不知道實際跑起來長什麼樣。
→ **我的工作是打開瀏覽器，把畫面翻譯回文字給 AI**。

### 6.2 會漏掉「一致性」

例：我加完 Reflection 頁之後，footer 和首頁都加了連結，但 AI 忘了加到 nav bar。
→ **我的工作是在多處出現的東西（nav / footer / sitemap）做整體檢查**。

### 6.3 會過度擴展

AI 有時候會主動寫一堆我沒要求的東西（多餘的 abstraction、過度的 error handling）。
→ **需要我說「夠了」或「不用這個」**。

### 6.4 不知道外部世界的即時狀態

「Vercel 最近被駭客入侵」這種新聞，AI 不一定知道。
→ **我的工作是把外部資訊輸入給它**。

## 七、可複製的協作模板

### Step 1：講目標，不講規格
❌「請幫我用 React useState + useContext 建立一個 tab 切換」
✅「做一個 Before/After 對比頁，三組設計決策」

### Step 2：看結果，不看程式碼
- 打開瀏覽器 / 裝置
- 看視覺、互動、文案
- 把問題描述成「使用者看到什麼不對」

### Step 3：給情報，不給指令
當 AI 的預設方案有問題時，不要直接說「改成 X」，而是：
- 告訴它「我遇到 Y 狀況」
- 讓它自己重新選擇方案

### Step 4：信任但驗證
AI 完成後問自己：
- 手機能用嗎？
- 無障礙有做嗎？
- nav 有連到新頁面嗎？
- 我會想把這個貼給朋友看嗎？

### Step 5：小步快跑
每做完一個功能就 commit + push，不累積太多未確認的變動。

## 八、這個協作模式適合誰？

### 適合
- 有設計直覺、但 code 能力有限的設計師
- 會看畫面判斷對錯的人
- 願意放權給 AI 做技術決定的人
- 知道自己要什麼（設計目標清晰）的人

### 不適合
- 想要 100% 掌控每一行 code 的人
- 沒有明確設計方向、想讓 AI 幫忙想設計的人（AI 可以實作，但需要你主導美學判斷）
- 不會跑瀏覽器預覽的人

## 十、對同事的三個建議

### 1. 不要把 AI 當成「自動補完」
把它當成一位初級工程師同事。給它任務、讓它提案、review 它的產出。

### 2. 保留「設計師判斷」的環節
AI 可以做出「能動的東西」，但「讓人感動的東西」還是要你來定義。

### 3. 學會用「現象」描述問題
你不需要會 debug，但需要會描述「看到什麼不對」。
這比學 JavaScript 更重要。

## 十一、結語

**這份 side project 不是 AI 幫我做的，是我和 AI 一起做的。**

AI 負責我不會的（寫 code、除錯、技術選型），我負責 AI 做不到的（美學判斷、使用者邏輯、方向決定）。

如果你是設計師，不要覺得「寫不出 code 就沒辦法做作品集」。你需要的是：
- 清晰的設計目標
- 看畫面判斷對錯的能力
- 把問題翻譯成自然語言的能力
- 一位會寫 code 的 AI 協作者
